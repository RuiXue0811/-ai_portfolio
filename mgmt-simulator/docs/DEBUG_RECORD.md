## 二、修改详情

### 修复1：logging 编码崩溃

**症状**：飞书 SDK 内部 `logger.debug()` 输出含中文 → Windows cp1252 编码器崩溃 → `on_message` 异常，回复发不出去。用户看到的就是"没反应"。

**修复**（`bot.py` 第7-14行）：

```python
# 重建 root logger 的 handler，使用 utf-8
for handler in logging.root.handlers[:]:
    logging.root.removeHandler(handler)
logging.basicConfig(
    level=logging.WARNING,
    format='[%(name)s] %(levelname)s: %(message)s',
    handlers=[logging.StreamHandler(sys.stdout)]
)
```

**要点**：`sys.stdout` 重编码只影响 `print()`，logging 模块有独立 handler，需要单独设置。

---

### 修复2：模型暴露身份

**症状**：回复中出现"我是Claude Code，由Anthropic开发的AI编码助手"等话术。

**根因**：原提示词是身份覆盖式（"你是人情世故助手"），Claude 模型会拒绝覆盖开发者设定，转而自我介绍。

**修复**（第44-56行）：改为**场景描述式**提示词：

```
【当前对话场景】
这是一场职场人际关系的分析讨论。用户正在向你咨询一个涉及职场政治、人际博弈的现实问题。

【回复要求】
- 每次只问一个引导性问题，帮用户自己看清局面
- 回复控制在 200 字以内
- 冷静、理性，像阅历丰富的导师
- 直接开始分析，不要开场白，不要提"作为AI"

【绝对禁止】
- 不要提你是 Claude 或任何 AI 模型
- 不要说"这不是我的专长"之类的话
```

---

### 修复3：消息去重

**症状**：手机端发"抽卡"，飞书重复推送 `on_message` 事件，消息发了2次。

**修复**（第35-36行、第172-176行、第196-200行）：

```python
processed_msg_ids = set()      # 全局去重
last_msg_time = {}             # 每个用户的上次消息时间
```

```python
# 消息 ID 去重
if msg_id in processed_msg_ids:
    return

# 2秒冷却
if user_id in last_msg_time and (now - last_msg_time[user_id]) < 2:
    return
```

**经验**：双保险——先后去重，防 set 清理后二次推送。

---

### 修复4：API 异常静默吞掉

**症状**：用户说"一直在分析"，等很久没反应。

**根因**：`on_message` 的 try/except 只打了日志，没发任何回复。API 失败后用户完全不知道发生什么。

**修复**（第137-146行）：

```python
try:
    reply = call_api(history)
except Exception as e:
    reply = "抱歉，我暂时无法连接到分析引擎。\n\n请稍后再试..."
```

**要点**：无论如何都要给用户回一句话，不能卡住。

---

### 修复5：对话历史被清空

**症状**：新回复跟上下文不连贯。

**根因**：检测到模型回复含"claude"/"anthropic"关键词后，执行 `conversations[user_id] = []` 清空历史，等于是让模型失忆后重新回答。

**修复**：删除了拒绝重试逻辑，不再清空历史。新版提示词已无暴露问题，检测逻辑保留但不做清除操作。

---

### 修复6：API 代理不稳定

**症状**：`SSLEOFError` / `Read timed out`。

**修复**（第114-145行）：

```python
def call_api(history, system_override=None):
    for attempt in range(3):
        try:
            resp = requests.post(..., timeout=30)
            ...
        except:
            if attempt < 2:
                time.sleep(2)
    raise Exception(...)  # 3次全失败
```

user_card_state = {}  # 抽卡交互状态

## 五、经验教训

1. **Windows 编码**：Python 默认 cp1252，logging、print、curl 都可能遇到中文编码问题。必须显式设置 utf-8。
2. **"没反应"的诊断顺序**：先看日志 → 确认消息是否收到 → 确认 API 是否返回 → 确认消息是否发出。不要猜权限或网络。
3. **模型身份**：Claude 模型不接受身份覆盖提示词，需要用场景描述替代"你是XXX"。
4. **消息去重双保险**：message_id + 时间冷却，缺一不可。
5. **API 异常处理**：每次 API 调用都必须是 try/except + 兜底回复，不能让用户干等。

## 六、cards.json 卡牌指代问题修复（2026-06-15）

### 问题背景

用户在体验"借他人背书加信"（card_020）时，机器人问"对方真正想要的是什么"，用户无法判断"对方"指的是谁，该卡场景中根本没有明确的"对方"存在。

检查全部 20 张卡牌后，发现共 6 张存在类似指代不清问题。

> **后续补充（2026-06-15）**：cards_021~120 批量生产后同样执行了清晰度审查，新 batch 发现 4 处问题已一并修复，见「七」节。

### 修复清单

| 卡牌 ID | 标题 | 问题位置 | 原文 | 修改为 |
|---|---|---|---|---|
| card_003 | 权力不对等时的迂回 | rule | "用同情心规避冲突" | "以你展现出的同理心和柔和姿态规避正面冲突" — 补充主语"你" |
| card_007 | 先发制人的反击 | scene + rule | scene 没有"盟友"角色；rule 引用"盟友"和"决策者" | scene 补入"信任的同事"角色；rule 将"决策者"改为"上级" |
| card_010 | 让越界有成本 | scene | "那个爱抢功劳的同事" 独立阅读时指代断层 | 加括号注明"上一张卡提到的，爱抢功劳的那位" |
| card_013 | 主动示弱的反向试探 | rule | "诱导对方的同情心" — "对方"实为读者自己（面试官） | 改为"诱导你（面试官）产生同情心" |
| card_015 | 不要被单独隔离 | scene | "咱们两位老板" 指代不明 | 改为"你我两位主谈" |
| card_020 | 借他人背书加信 | scene | 场景无"对方"，机器人提问框架失效 | 加入"赢得在场关键决策者的认可"，让"对方"有所指 |

### 根本原因

卡牌内容编写时默认读者能联系上下文脑补，但机器人提问模板是固定句式（"对方真正想要的是什么"），当卡牌场景中不存在明确对手/对方时，提问与场景脱节，用户无法理解。

### 经验

- 每张卡牌应独立自洽，"对方"必须能从本卡 scene 中找到对应角色
- rule 中引入的角色（如"盟友"、"决策者"）必须先在 scene 中出场
- 读者视角（你）和场景里的"对方"不能混用同一个词

## 八、card_118 类型修正（2026-06-15）

### 问题

同步检查发现卡牌类型分布：识局×30、**读人×29**、防身×30、**应对×31**，总计 120 张但分布不均。

### 根因排查

逐一审查 31 张应对卡，确认没有读人卡被误标为应对。数量偏差来源于批量生产时分配不均，并非分类错误。

但发现 `card_118`「在对方的框架里找漏洞」被标为应对，实际不符：
- **场景核心**：识别对方逻辑里的隐藏前提假设
- **技能本质**：读懂对方的论证结构，而非行动策略
- **应对特征缺失**：rule 没有"怎么做"的行动指令，而是"怎么看"的识别方法

### 修复

将 `card_118` type 从 `应对` 改为 `读人`。修正后分布：识局×30、读人×30、防身×30、应对×30。

### 经验

类型归属判断标准：
- **识局**：识别整体局面和态势
- **读人**：识别具体人物的动机、意图、行为模式
- **防身**：保护自己、避免被动
- **应对**：主动采取行动改变局面

rule 里如果核心动词是"识别/看清/理解"→ 读人；如果是"做/说/争取/推进"→ 应对。
