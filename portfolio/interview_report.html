S — Situation

背景：信贷运营的三个结构性痛点

📉 复贷率低
存量用户结清后沉默，缺乏有效的个性化触达机制，复贷转化靠人工跟进，规模化困难
⏱️ 运营效率低
不同用户类型需手动分版本写话术，一个运营每天只能处理有限用户，无法覆盖全量存量池
⚠️ 合规风险高
金融广告话术极易触碰银保监红线（如"保证放款""零门槛"），违规后果严重，靠人工审核漏洞大
我有 3 年信贷运营经验，深知这条链路的问题：分层靠人工、话术靠感觉、合规靠运气。 这个项目的出发点，是把我脑子里的 SOP 系统化、自动化，让它能跑起来、能复用、能扩展。
T — Task

目标：把信贷 SOP 跑进 AI 工作流

🎯 核心链路目标
实现从"输入用户数据"到"输出可用触达方案"的全自动化——
用户分层 → 知识库检索 → 话术生成 → 合规审查 → 触达策略建议，全程无需人工干预

③ 输出格式要结构化，让运营可以直接使用，无需二次处理

排障过程：三个核心 Bug 的根因与修复

B1
合规节点 3/3 全部失败（通过率 0%）
根因：Prompt 只列了规则文字，LLM 把 warning 级别的用词也判成了 FAIL，且不输出二值结果。
规则文字对 LLM 的约束力远弱于具体示例。
修复：加入 4 个 PASS/FAIL 对照示例 + 显式要求输出二值判断 → 通过率 0% → 89%
B2
模板变量字面量泄露（输出 {{days_since_claer}}）
根因：变量名拼错（claer → clear），Coze 变量绑定严格匹配，拼错后静默降级为字面量字符串，不报错。
无代码平台的变量绑定是隐式的，比代码更难 debug。
修复：纠正变量名 + 改从节点"输入"区接收字段，不用双花括号写法
B3
用户 U_C003（days=75）分到 A 层，应为 C 层
根因：LLM 数值比较存在盲区，75 > 30 但它判成了 A 层条件。
数值比较不该交给 LLM，这是硬规则，LLM 做的是"理解"不是"计算"。
修复：加入「D → A → B → C → 兜底」命中即停的判定顺序 + 把 75 天反例写入 prompt → 分层准确率 100%
Phase 1 核心洞察
这三个 Bug 让我总结出一条原则：能用代码做的硬规则，绝不交给 LLM。 LLM 的价值在于理解语义、生成内容，而不是做精确计算或严格匹配。 把它用在错误的地方，是不稳定性的根源。
A — Action

第二阶段：升级为 Claude Code Skill

Phase 2 · Claude Code Skill 重构

Coze 原型验证了业务逻辑可行，但暴露了平台本身的三个瓶颈，促使我做架构升级。

维度 Coze 的瓶颈 Skill 的解法
分层准确性 依赖 LLM 判断数值，B3 Bug 根源在此 改写为确定性 if/else 代码，从根本上消除 LLM 数值盲区
合规审查质量 Coze 内置模型语义理解有限，需大量 few-shot 才能到 89% 接入 Claude API，语义理解能力大幅提升，合规通过率提升至 95%+
批量处理 单次只能处理一条，批量需付费升级 Coze 计划 原生支持多用户并行处理，一次输入 CSV 跑全量存量池
知识库维护 Coze 知识库 UI 操作，版本管理不透明 合规话术库以本地文件存储，可 Git 版本管理，改动有记录
可复用性 换数据要重新配置节点参数 Skill 接受结构化输入，换数据直接跑，零配置成本

Skill 架构设计：三个关键决策

决策 1：分层逻辑代码化
将 A/B/C/D 分层规则写成纯 JavaScript 函数，输入 days_since_clear 和行为标签， 输出层级字符串。

确定性代码替代 LLM 判断，分层准确率从"依赖 prompt 质量"变成 100% 可保证
决策 2：合规审查 Claude 化
合规审查任务交给 Claude API，用结构化 Prompt 传入 7 类红线词规则 + PASS/FAIL 示例。

Claude 的语义理解更强，能识别隐式违规（如"基本上都能通过"）而不只是关键词匹配
决策 3：批量并行处理
Skill 接收用户数据数组，对每条记录并行调用 AI 节点， 最终输出结构化 JSON，方便下游系统直接消费。

100 个用户从顺序跑的 25 分钟，降到并行的 约 2 分钟

L — Learnings

三条可迁移的经验

① LLM 的适用边界
LLM 擅长语义理解和内容生成，不适合做精确数值比较和严格规则匹配。 B3 Bug 就是把错误任务交给了 LLM。 识别这条边界，是 Prompt Engineering 里最重要的判断力。
② 示例 > 规则文字
在合规审查里，把规则写成文字描述效果差， 加入 PASS/FAIL 对照示例后通过率从 0% 到 89%。 LLM 是通过模式匹配学习的，具体示例比抽象规则对它更有效。
③ 先 MVP 后重构
用 Coze 快速出原型，找到真实痛点（B1/B2/B3），再针对性升级架构。 不是一开始就写代码，而是用最低成本先验证业务逻辑。 这个顺序让后期的重构更有方向。
